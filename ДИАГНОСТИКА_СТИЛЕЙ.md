# Диагностика проблемы со стилями

## Проблема
В логах видно:
- `cart.css` обрабатывается успешно (view вызывается)
- `style.css` и `main.js` НЕ обрабатываются (view НЕ вызывается, только `WARNING log Not Found`)

## Что это означает
Если в логах НЕТ сообщения `=== STATIC FILE REQUEST ===`, значит запрос НЕ доходит до Django view `serve_static_file`.

Возможные причины:
1. Запросы перехватываются до Django (nginx, Passenger)
2. Файлы не существуют (но вы сказали, что они есть)
3. Проблема с кешированием браузера

## Диагностика на сервере

### Шаг 1: Проверьте, что файлы действительно существуют

```bash
cd ~/onesimus/onesimus

# Проверьте CSS
ls -la staticfiles/css/style.css
ls -la staticfiles/css/cart.css

# Проверьте JS
ls -la staticfiles/js/main.js

# Проверьте размеры файлов
du -h staticfiles/css/style.css
du -h staticfiles/css/cart.css
du -h staticfiles/js/main.js
```

**Если файлов нет** - соберите статику:
```bash
python manage.py collectstatic --noinput
```

### Шаг 2: Проверьте права доступа

```bash
# Проверьте права
ls -la staticfiles/css/
ls -la staticfiles/js/

# Убедитесь, что файлы читаемые
chmod 644 staticfiles/css/*.css
chmod 644 staticfiles/js/*.js
```

### Шаг 3: Проверьте через curl напрямую

```bash
# Проверьте, что возвращает сервер
curl -v https://onesimus25.ru/static/css/style.css 2>&1 | head -30

# Сравните с cart.css (который работает)
curl -v https://onesimus25.ru/static/css/cart.css 2>&1 | head -30
```

**Обратите внимание на:**
- HTTP статус (200 или 404?)
- Content-Type (правильный ли?)
- Размер ответа (совпадает ли с размером файла?)

### Шаг 4: Проверьте логи более детально

```bash
# Смотрите все запросы к статике
tail -100 logs/django.log | grep -E "(static|STATIC)"

# Ищите конкретно style.css
tail -100 logs/django.log | grep -i "style.css"

# Ищите конкретно main.js
tail -100 logs/django.log | grep -i "main.js"
```

**Если видите `=== STATIC FILE REQUEST ===`** - view вызывается, но файл не находится.
**Если НЕ видите** - view не вызывается, запрос перехватывается до Django.

### Шаг 5: Проверьте, что паттерн для статики действительно первый

```bash
# Проверьте порядок URL паттернов в коде
grep -A 5 "urlpatterns = \[" config/urls.py
grep -A 5 "re_path.*static" config/urls.py
```

### Шаг 6: Проверьте, нет ли проблем с кешированием

Попробуйте запросить файлы с уникальным параметром:

```bash
# Запросите с параметром версии
curl -I "https://onesimus25.ru/static/css/style.css?v=test123"

# Сравните с cart.css
curl -I "https://onesimus25.ru/static/css/cart.css?v=test123"
```

### Шаг 7: Проверьте, что DEBUG действительно False

```bash
grep "^DEBUG" config/settings.py
```

Если `DEBUG = True`, Django использует другой механизм (`staticfiles_urlpatterns`), а не наш кастомный view.

## Возможные решения

### Решение 1: Если файлы не существуют

```bash
python manage.py collectstatic --noinput
```

### Решение 2: Если view не вызывается

Попробуйте временно установить `DEBUG = True` для тестирования:

```bash
# Временно включите DEBUG
sed -i "s/DEBUG = False/DEBUG = True/" config/settings.py

# Перезапустите
touch tmp/restart.txt

# Проверьте
curl -I https://onesimus25.ru/static/css/style.css

# Если работает - проблема в кастомном view при DEBUG=False
# Если не работает - проблема в другом месте
```

### Решение 3: Настройте nginx (рекомендуется)

Если Django view не работает стабильно, настройте nginx:

```bash
cd ~/onesimus/onesimus
mkdir -p .nginx

STATIC_PATH=$(realpath staticfiles)
cat > .nginx/onesimus25.ru.conf << EOF
location /static/ {
    alias $STATIC_PATH/;
    expires 30d;
    add_header Cache-Control "public, immutable";
    access_log off;
    
    types {
        text/css css;
        application/javascript js;
        image/png png;
        image/jpeg jpg jpeg;
        image/svg+xml svg;
        image/webp webp;
    }
    default_type application/octet-stream;
}
EOF

cp .nginx/onesimus25.ru.conf .nginx/www.onesimus25.ru.conf
touch tmp/restart.txt
```
