# Решение конфликта миграций

## Проблема

На сервере уже есть миграция `0012_synclog_productcharacteristic`, которая включает обе модели (`SyncLog` и `ProductCharacteristic`). Локально мы создали файл `0012_synclog.py`, который также включает обе модели, что создает конфликт.

## Решение

### Вариант 1: Заменить файл на сервере (если миграция еще не применена)

Если миграция `0012_synclog_productcharacteristic` на сервере еще **не применена**, просто замените файл:

```bash
cd /home/o/onesim8n/onesimus/onesimus
# Удалите старый файл
rm catalog/migrations/0012_synclog.py
# Загрузите новый файл 0012_synclog_productcharacteristic.py
```

Затем примените миграцию:
```bash
python manage.py migrate
```

### Вариант 2: Создать merge-миграцию (если миграция уже применена)

Если миграция на сервере **уже применена**, нужно создать merge-миграцию:

```bash
cd /home/o/onesim8n/onesimus/onesimus
source venv/bin/activate
python manage.py makemigrations --merge catalog
```

Django создаст файл типа `0013_merge_xxxxx.py`, который объединит обе ветки.

Затем примените миграцию:
```bash
python manage.py migrate
```

### Вариант 3: Удалить локальный файл и использовать серверный

Если на сервере файл уже правильный и применен, просто удалите локальный конфликтующий файл:

```bash
# Локально (на вашем компьютере)
rm catalog/migrations/0012_synclog.py
```

И используйте файл с сервера `0012_synclog_productcharacteristic.py`.

## Рекомендуемый подход

1. **Проверьте на сервере, применена ли миграция:**
   ```bash
   python manage.py showmigrations catalog
   ```

2. **Если миграция НЕ применена (нет галочки):**
   - Удалите `0012_synclog.py` на сервере
   - Загрузите `0012_synclog_productcharacteristic.py`
   - Примените: `python manage.py migrate`

3. **Если миграция УЖЕ применена (есть галочка):**
   - Создайте merge-миграцию: `python manage.py makemigrations --merge catalog`
   - Примените: `python manage.py migrate`

- ✅ Удален файл `catalog/migrations/0012_synclog.py` (конфликтующий)

Теперь нужно синхронизировать с сервером.
